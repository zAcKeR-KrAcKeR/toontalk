# ToonTalk — Full Project Context
> Last updated: 2026-05-17 09:32 IST

## 🎯 What We Are Building

**ToonTalk** — an interactive cartoon companion. A child watches a Hindi cartoon episode (Shinchan, Doraemon) in a browser. They press a mic button and ask a question in Hindi. The app detects which character is on screen, generates an in-character Hindi answer, and **plays it back in that character's actual cloned voice extracted directly from the video**.

### Stack
| Layer | Technology |
|---|---|
| Backend | FastAPI + Python 3.10 |
| Video Intelligence | VideoDB SDK (scene indexing, spoken word index, voice generation) |
| VLM | Gemma 4 31B (via VideoDB sandbox) |
| Transcription | Groq Whisper-large-v3-turbo |
| LLM | GPT-4o via VideoDB OpenAI-compatible proxy |
| Voice Cloning | ElevenLabs via VideoDB proxy (`model_name="elevenlabs"`, `config={"ref_audio": url}`) |
| Frontend | React (Vite) |
| Audio Extraction | ffmpeg (local) |

## ✅ What Works (Current State)

### 1. Auto Character Detection from Video
- The VLM (Gemma 4 31B) watches every 10s frame and outputs structured JSON per scene:
  ```json
  {"characters": [{"name": "Shinchan", "speaking": true, "emotion": "happy"}, ...], ...}
  ```
- `detect_characters_from_scene_index()` in `videodb_service.py` parses all scenes, maps names via `ALIAS_TO_CHARACTER` registry in `character_config.py`, and returns character keys like `["shinchan", "misae", "hiroshi", "kazama", "nene", "masao", "bo"]`.
- **7 characters auto-detected** from the test video: `bo, gian, hiroshi, masao, misae, nene, shinchan`

### 2. Voice Extraction from Video (ffmpeg pipeline)
- For each detected character:
  1. **VLM path (primary):** Find a scene where VLM tagged `speaking: true` for that character → get scene start/end timestamps
  2. **Spoken word index fallback:** Search using Hindi name keywords (`शिनचान`, `हिरोशी`, etc.) → get dialogue segment timestamps
  3. **Last resort:** Use first 30s of video
- `video.generate_stream(timeline=[(start, end)])` → HLS `.m3u8` URL
- `ffmpeg -i <m3u8> -vn -acodec libmp3lame -ar 22050 -ab 64k → temp.mp3`
- `coll.upload(file_path=tmp.mp3, media_type="audio")` → `ref_audio_url`
- Saved immediately to local JSON cache — **no validation step** (lazy on first TTS use)
- ⏱ ~30s per character

### 3. Voice Cloning TTS (ElevenLabs via VideoDB proxy)
- **Confirmed working:** `coll.generate_voice(text=..., model_name="elevenlabs", config={"ref_audio": ref_audio_url})` **actually clones the voice** using the extracted audio segment
- Returns an `Audio` object directly (no job polling needed)
- ⏱ ~1-2 seconds per TTS call
- No sandbox required — instant

### 4. Q&A Pipeline (`/api/ask`)
- Parallel: scene context + character detection run simultaneously
- Character detected by keyword match first, LLM fallback if ambiguous
- Answer generated by GPT-4o in-character Hindi, max 10 words (1 sentence for speed)
- Frontend calls `/api/generate-tts` after receiving text to play audio
- Total latency: ~3-5s text + ~2s audio

### 5. Processing Pipeline (`/api/process/{video_id}`)
Full pipeline when a new video is uploaded:
1. `index_scenes()` — VLM scene tagging
2. `index_spoken_words()` — Whisper transcription
3. `detect_characters_from_scene_index()` — auto-detect who's in the video
4. `clone_character_voice()` × N — extract + upload ref audio for each character

### 6. Character Registry (`character_config.py`)
- Full Shinchan universe: `shinchan, misae, hiroshi, nene, kazama, masao, bo`
- Full Doraemon universe: `doraemon, nobita, shizuka, gian, suneo`
- Each character has: `hindi_name`, `description`, `system_prompt`, `greeting`, `voice_search_keywords`
- `ALIAS_TO_CHARACTER` dict maps 100+ name variants (Hindi, English, alternate spellings) → character key

## ❌ What Failed and Why

### Failure 1: `coll.upload(url=stream_url)` — "Download failed"
**What happened:** We tried to re-upload the HLS stream URL directly to VideoDB to create an Audio asset for OmniVoice.

**Why it failed:** `video.generate_stream(timeline=...)` returns an HLS `.m3u8` playlist URL (protected). VideoDB's upload endpoint cannot fetch/parse `.m3u8` playlists — it needs a flat downloadable file.

**Fix:** Added ffmpeg step: download HLS → local MP3 → upload the MP3 file. (`subprocess.run(["ffmpeg", "-i", m3u8_url, ..., "tmp.mp3"])` + `coll.upload(file_path=tmp.mp3)`)

---

### Failure 2: OmniVoice validation timeout (300s → killed)
**What happened:** After uploading ref audio, we ran a test voice generation with OmniVoice to "validate" the clone before saving.

**Why it failed:** OmniVoice runs on a sandbox that takes 3-5 minutes to provision on cold start. With 7 characters × 5 minutes = 35+ minutes of blocking. Job kept timing out at 300s.

**Fix 1 (wrong):** Raised timeout to 600s — still blocked.

**Fix 2 (wrong):** Tried `sandbox_task.cancel()` + `await sandbox_task` — the sandbox task was wrapping a blocking thread executor call so `cancel()` had no effect. Caused an infinite hang.

**Fix 3 (wrong):** `sandbox_task.cancel()` fire-and-forget — but `asyncio.ensure_future(get_sandbox_id())` still kept the sandbox thread running and blocking the event loop.

**Final Fix (correct):** Removed the entire OmniVoice validation step. The clone is saved as soon as ref_audio is uploaded. Validation happens lazily on first TTS call.

---

### Failure 3: OmniVoice sandbox blocking TTS calls
**What happened:** On every `/api/generate-tts` call, we awaited `get_sandbox_id()` which provisions a new medium sandbox — taking 3-5 minutes on cold start.

**Root discovery:** During testing, we found `coll.generate_voice(model_name="elevenlabs", config={"ref_audio": url})` **actually performs voice cloning** via the ElevenLabs model, returns instantly (~1s), and requires **no sandbox at all**.

**Final Fix:** Replaced OmniVoice TTS path entirely with ElevenLabs + ref_audio. OmniVoice is no longer used for TTS.

---

### Failure 4: All characters getting the same voice segment `[650s-680s]`
**What happened:** The fallback spoken-word search used generic queries like `"bolna"` and `"speaking"` for all characters. All queries returned the same high-activity scene at 650s.

**Fix:** Added character-specific Hindi search queries for all 7 characters:
- `shinchan` → `["शिनचान", "shinchain", "shin chan", "mama", ...]`
- `gian` → `["जियान", "gian", "takeshi", "singing", ...]`
- `hiroshi` → `["हिरोशी", "hiroshi", "papa", ...]`
- etc.

---

### Failure 5: VLM `speaking: true` tag not found for most characters
**What happened:** The VLM scene prompt asks it to set `speaking: true` when a character speaks. In practice, the VLM sometimes forgets this or marks it inconsistently.

**Current state:** All 7 characters fell through to the spoken-word fallback. The VLM path is correct code — it's a VLM output quality issue.

**Mitigation:** Fallback to spoken-word search with character-specific Hindi queries works well enough for now.

---

### Failure 6: Backend server dying between sessions
**What happened:** Multiple times the server process was killed (user cancelled terminal, watchfiles reload, hung tasks). The voice cloning jobs were lost mid-way.

**Fix:** The metadata JSON is saved per-character as each clone completes. If the server dies, re-running `/api/clone-voices` skips already-cloned characters (`if character_name in meta.get("character_voices", {})` early-return).

## 🔄 Current Flow (End-to-End)

```
1. Upload video via UI
        ↓
2. POST /api/process/{video_id}
   └─ VLM scene index (Gemma 4 31B, every 10s frame)
   └─ Whisper spoken word index (Groq)
   └─ Auto-detect characters from scene JSON
   └─ For each character:
        └─ Find speaking segment (VLM tags → fallback spoken word search)
        └─ ffmpeg: HLS m3u8 → local MP3
        └─ Upload MP3 → ref_audio_url saved to JSON
        ↓
3. Watch & Chat: User presses mic
        ↓
4. POST /api/ask-voice (audio blob)
   └─ Groq Whisper → Hindi question text
   └─ Parallel: scene context + character detection
   └─ GPT-4o → 1-sentence Hindi answer (in character)
        ↓
5. POST /api/generate-tts
   └─ ElevenLabs + ref_audio_url → cloned voice audio (~1-2s)
   └─ Frontend plays audio + shows manga bubble
```

## ⚠️ Known Issues / Next Steps

1. **URL expiry:** `ref_audio_url` from `audio.generate_url()` is a signed GCS URL that expires in ~7 days. If the server is restarted after 7 days, TTS will fail. Fix: store `ref_audio_id` and call `generate_url()` fresh on each TTS call instead of caching the URL.

2. **ElevenLabs clone quality:** The voice clone quality depends on the 30s audio segment extracted. Background music or multiple speakers in the clip will degrade quality. Best practice: find a clean solo-speech scene.

3. **VLM `speaking: true` reliability:** The Gemma 4 31B model sometimes doesn't set this flag. The primary detection path (VLM) works 0/7 times on the current video — all fall to spoken-word fallback. Could be improved with a more explicit prompt.

4. **Sequential cloning:** Characters are cloned one at a time (~30s each). 7 characters = ~3.5 minutes total. Could be parallelized with `asyncio.gather()`.

5. **Frontend has no progress bar for voice cloning.** The Library UI shows "Clone Voices" button but doesn't stream real-time status per-character.
